   Hi again Aaron, this is the second of three posts about analyzing msfvenom shellcode.  This one involves read_file.  Msfvenom is a command line tool by Rapid7 that generates payloads in various formats and encoding schemes.  According to Offensive Security, it combined functionally of both msfpayload and msfencode as of June 8, 2015 by placing both tools under a single framework.


Getting Started

   Using any machine with Metasploit Framework, dump available payloads to verify the existence of ../read_file.  The specific command is msfvenom -l payloads | grep "linux/x86":


Next step is to determine what the payload requires.  Dump the summary information with the command msfvenom -l linux/x86/read_file --list-options.  Note that the basic options take priority and one option, the PATH, must be set in the payload.  According to the description, read_file reads a file from PATH and writes it out to a file descriptor.



Dumping the Disassembly

   Dump the disassembly with ndisasm using command msfvenom -p linux/x86/read_file PATH=kingofthecastle.txt R | ndisasm -u -.  Ensure PATH is set or create a simple text file to read in the payload (shown below).  The -p option specifies the payload.  R specifies raw output, which is then piped into ndisasm.   ndisasm stands for "Netwide Disassembler" and is the companion disassembler for the nasm assembler used in the previous four blog posts.  The -u option specifies 32-bit mode.  Finally the single dash is a common UNIX option which allows ndisasm to allow input from stdin.

Libemu Analysis

   Libemu is a tricky tool to install so most of its analysis is done on the web VM.  A successful install comes from this repository as it is the most recently supported version.  Note that in the repository's installation, tools are already included in the path so there is no need to search for binaries in /tools/.  The only Libemu tool missing from the VM is dot, which allows visual analysis via the creation of png files.  Unfortunately dot cannot be used because the sctest output is incomplete.
   Run the below command to pipe raw shellcode into sctest.  Once again sctest stops after the first syscall, thus a proper Control Flow Graph (CFG) file cannot be made using the -G option.  Proper functionality was tested on a Libemu install by running sctest -vvv -Sc 100000 -G bind-shell.dot on the linux/x86/shell_bind_tcp raw payload.  A png file depicting the payload's control flow was successfully created with the dot tool.  The read_file payload provides no additional analysis vectors with Libemu.



GDB Analysis

   Once again, analysis is restricted to dynamic analysis in GDB and manual analysis with ndisasm.  In order to analyze the shellcode in GDB, it will need to be extracted and placed into a C-code harness.  Use the below command to create a payload in C-code format.  To list additional formats, simply run msfvenom --list formats.


Copy and paste shellcode into the C-code harness.


Save the file and compile it with gcc -fno-stack-protector -z execstack shellcode.c -o shellcode.  Start GDB in quiet mode, set the disassembly flavor to intel, and disassemble the main() function.  Stop at instruction call eax.  This is where the shellcode is passed control.


Step one more instruction then disassemble the payload.  This should look identical to the nasm disassembly.


With offsets clearly expressed above, the payload makes much more sense.  A jump-call-pop technique is used when following the first few instructions.  Instead of popping garbage instructions into ecx, their address is popped into ebx.
   Once again, system calls (syscalls) can be tracked in the disassembly by noting their eax values and referring to the table.  In order, the syscalls are as follows:

  • open()
  • read()
  • write()
  • exit()
    • Before calling open(), a jump-call-pop is done to place the payload's address into ebx and 0x5 into eax.  After jumping to the call instruction, the address of the following instruction is outlined below and will be pushed onto the stack after executing call 0x56557022.  The corresponding hook-stop is defined below: 

    										
(gdb) define hook-stop
Type commands for definition of "hook-stop".
End with a line saying just "end".
>print/x $eax
>print/x $ebx
>print/x $ecx
>print/x $edx
>disassemble $eip,+28
>x/24xb $esp
>end
(gdb)

    Step again to observe 0x5655705d pushed onto the stack.


    Step through the remaining instructions up to the first syscall.  Based on the open() man page, the function takes two parameters: pathname to the requested file as a pointer, and flags.  This should look familiar to what was observed in adduser.  With this in mind, dump the string contents at address 0x5655705d (or ebx) to reveal the name of the requested file!


    The next syscall is to open() shown by loading 0x5 into eax.  It takes two parameters: pathname to the requested file as a pointer, and flags.  Revisit the DWORDs pushed onto the stack and convert the bytes to ascii.  It spells /etc/passwd!  The values placed into ecx are 0x0401, which represent file permissions similar to what is observed in chmod.  The flag represents both O_WRONLY (write only) and O_NOCTTY (do not assign controlling terminal).  Their exact values can be found in /usr/include/asm-generic/fcntl.h.  If successful, the return value is a file descriptor used in later syscalls.


    Step through the open() syscall to receive a file descriptor value of 8.  


    The next branch uses a strange call instruction that points to the middle of a later instruction below.  Specifically it points to the middle of instruction or bl, BYTE PTR [ecx-0x75].  This seems out of place.  Why would a program do this?  Remember that the call instruction places the address of the next instruction (0x5655704b) onto the stack.


    After the call instruction, a pop ecx instruction appears!  This activity appears very similar to that found in JMP-CALL-POP, which means garbage instructions starting at address 0x5655704b represent a "sub-payload" inside the adduser payload!  It is likely a parameter to a later system call.
       The subsequent instruction changes to mov edx, DWORD PTR [ecx-0x4] and 0x5655704b appears on the stack.  Based on the call, it appears changes occured because execution control was passed to the middle of an instruction; reshuffling opcode bytes in the process.


    Verify the opcode bytes of mov edx, DWORD PTR [ecx-0x4] by dumping it into shell-storm's online assembler/disassembler tool.  


    The output matches the next 3 opcode bytes from the ndisasm disassembly.  Below is the snipped output (before executing call 0x53) to show relevant sections of the payload.  Byte 0x59 is indeed followed by opcode bytes 0x8b, 0x51, and 0xfc.

0000002B  6D                insd

; snip garbage instructions / payload

    Step through mov edx, DWORD PTR [ecx-0x4] to load 0x21 into edx.  According to the write() man page, edx holds the number of bytes to write; 33 in this case.  Register ecx contains the address holding the contents to write; in this case, it's the starting point of the 33 bytes to write.  The file descriptor is loaded into ebx and the system call number, 0x4, is loaded into eax.


    A successful execution of write() returns the number of bytes written.  33 bytes were successfully written according to the eax value.  All bytes of the sub-payload were written because 0x5655704b + 0x21 = 0x5655706c, which is the start of instruction pop ecx!


    The contents written to /etc/passwd can be checked by dumping 33 bytes.  Below shows the string of user bob, their password, User ID (UID), Group ID (GID), home directory, and starting program (typically an interactive shell).



    Thoughts and Lessons Learned

       Analyzing payloads from msfvenom is great way to familiarize with other types of shellcode.  Not all exploits involve reverse shells!  It is also important to understand not every tool will yield fruitful analysis.  Libemu provided little help with this analysis but is much friendlier towards shell-type payloads.  The next msfvenom analysis involves read_file.
