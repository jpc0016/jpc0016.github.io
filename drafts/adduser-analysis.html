   Good evening Tim, this post involves analyzing the adduser shellcode sample from msfvenom.  Msfvenom is a command line tool by Rapid7 that generates payloads in various formats and encoding schemes.  According to Offensive Security, it combined functionally of both msfpayload and msfencode as of June 8, 2015 and placed both tools under a single framework.


Getting Started

   Using any machine with Metasploit Framework, dump available payloads and verify ../adduser exists.  The specific command is msfvenom -l payloads | grep "linux/x86":


Next step is to determine what the payload requires.  Dump the summary information with the command msfvenom -l linux/x86/adduser --list-options.  Note that the basic options take priority and in this case, advanced options are taken care of.  According to the summary, the adduser payload creates a user, metasploit, with password, metasploit, and adds it to the admin group.



Dumping the Disassembly

The above dump is the entire assembly output, which includes multiple calls to int 0x80.   Multiple DWORDs with ascii characters are also pushed onto the stack.  This information is useful; however, it is difficult to parse on its own.  Further analysis is needed starting with Libemu.


Libemu Analysis

   Libemu is a tricky tool to install so most of its analysis is done on the web VM.  The only Libemu tool missing from the VM is dot, which allows visual analysis via the creation of png files.  Unfortunately dot cannot be used because the sctest output is incomplete.  Navigate to the ../tools/sctest folder to find the sctest binary.


After navigating to the folder, run the following command to pipe raw shellcode into sctest.  For some reason the output stops at the first syscall, thus a proper Control Flow Graph (CFG) file cannot be made using the -G option.  Proper functionality was tested on a Libemu install by running sctest -vvv -Sc 100000 -G bind-shell.dot on the linux/x86/shell_bind_tcp raw payload.  A png file depicting the payload's control flow was successfully created with the dot tool.  The adduser payload provides no further analysis vectors with Libemu.



GDB Analysis

   In order to analyze the shellcode in GDB, it will need to be extracted and placed into a C-code harness.  Use the below command to create a payload in C-code format.  To list additional formats, simply run msfvenom --list formats.


Copy and paste shellcode into the C-code harness.


Save the file and compile it with gcc -fno-stack-protector -z execstack shellcode.c -o shellcode.  Start GDB in quiet mode and disassemble the main() function.  Stop at instruction call eax.  This is where the shellcode is passed control.


Step one more instruction then disassemble the payload.  This should look familiar to the nasm disassembly.


The above assembly includes 4 calls to int 0x80 and multiple ascii bytes pushed onto the stack.  By noting the values of eax, each syscall can be mapped in order using the syscall table: setreuid(), open(), write(), and exit().


Starting with setreuid(), 0x46 (70) is loaded into eax while ebx and ecx are set to zero.  The parameters are zero because the shellcode sets the UID to zero according to the earlier description.  Step through the first few instructions up to the syscall to confirm the parameters.  The corresponding hook-stop is defined below: 

										
(gdb) define hook-stop
Type commands for definition of "hook-stop".
End with a line saying just "end".
>print/x $eax
>print/x $ebx
>print/x $ecx
>print/x $edx
>disassemble $eip,+28
>x/24xb $esp
>end
(gdb)

Step again to observe results from the call to setreuid().  Register eax is zero so the call was successful.


The next syscall is to open().  It takes two parameters: pathname to the requested file, and flags.  Revisit the DWORDs pushed onto the stack and convert the bytes to ascii.  It spells /etc/passwd!  The values placed into ecx are 0x0401, which represent file permissions similar to what is observed in chmod.  The flag represents both O_WRONLY (write only) and O_NOCTTY (do not assign controlling terminal).  Their exact values can be found in /usr/include/asm-generic/fcntl.h.  If successful, the return value is a file descriptor used in later syscalls.
