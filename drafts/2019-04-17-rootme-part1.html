Root-me Cracking Challenges (Part 1)

This is part one of a series in reverse engineering binaries provided by root-me.org. Root-me is a French-based computer security community that emphasizes hands-on learning. I highly recommend creating an account for practice. If you are new to the field, you should read their Frequently Asked Questions (FAQ) page for a "learning path." It includes links to textbooks, research papers, and relevant challenges.

Since this is part 1 of the cracking challenges, a brief primer on binary analysis is needed. The purpose of binary analysis is to explore the behavior of executable files when source code may not be available. This is typically done at the assembly code level but can involve Intermediate Representations (IRs) on a tool-by-tool basis. Most Linux operating systems ship with basic command-line tools such as file, readelf, objdump, and xxd for analyzing binaries. The basic GNU debugger tool also comes default with Linux. Some questions can be answered with rudimentary knowledge of assembly language and decompilation; however, more advanced programs such as self-modifying malware require in-depth knowledge of classes and structures. For now we will analyze beginner files.

To start from the root-me homepage, click on Challenges >> Cracking


You are brought to a list of binary cracking challenges ordered by increasing difficulty. Select the first binary file labeled ELF-0 protection. On the next page click Start the challenge to download the file as ch1.zip and unzip to ch1.bin.



If not already done, ensure you have a software reverse engineering tool installed to aid your cracking. You can elect to use the default GNU tools, but I prefer a visual tool suite for solving these files. You will need all the help you can get while cracking more difficult binaries. This isn't to say objdump and file are useless. There is definitely a place for them and are great for initial "quick looks" as well as aiding dynamic analysis. Speaking of quick-look, we can run the binary file to check for expected input values. I received the input below when running ch1.bin, which translates to "please enter your password" in French.

As expected, I got the wrong answer as indicated by the phrase "too bad, try again" in French. The good news is I have strings to look for while dissecting the binary. These are typically stored in the .rodata section. At this point we can proceed with our reverse engineering tools to analyze the behavior.

I will use Ghidra to disassemble ch1.bin. Ghidra is an open-source reverse engineering framework developed by the National Security Agency (NSA). It provides support for a multitude of software architectures and instruction sets and ships with a wide array of analysis features. Ghidra can be downloaded here and requires Java Development Kit (JDK) version 11+ which can be downloaded here. Installation is pretty straight-forward once Java is set up. Unzip the ghidra package, navigate to the root folder (typically cd ghidra_ [$version-number]_PUBLIC_[$date-of-release]/ghidra_[$version-number]/), and run:

sudo ./ghidraRun.

[insert command line picture here]

You will run ghidraRun.bat if installing Ghidra on a Windows system. When you first run this command, Ghidra will ask you for the specific path to the JDK v11+ folder. Enter the folder path for Ghidra to complete startup. The first window that appears is a project window with no active project.



This is Ghidra's home window from which you can use various tools. Ghidra ships with CodeBrowser and Version Tracking tools by default. A motivating factor for open-sourcing Ghidra was for the development community to fill the Tool Chest and effectively reduce the NSA's workload (and identify talent for recruitment). My windows appear black because I changed the color theme. This can be configured by going to Edit >> Tool Options... >> Tool , checking Use Inverted Colors, then restarting Ghidra.

The next step is to create a new project by selecting File >> New Project... Make it a Non-Shared Project then give it a home directory and name. Ghidra saves the project file using a .gpr extension. When finished, an empty project will appear in the project window; however, the project has no file. Import a file to analyze using File >> Import File... or simply enter the "I" key. Navigate to where you saved ch1.bin and import it.


Ghidra automatically detects the file as ELF executable and defaults to the gcc compiler. Enter the destination folder and program name if you wish to change. Default options are usually sufficient in this window. An import summary window will subsequently appear. This shows various details such as file size, executable format, required libraries, and others. Some of this information can be obtained by running readelf -h ch1.bin. Click OK then drag and drop ch1.bin to the green dragon icon to open CodeBrowser. The green dragon will enlarge and fade into the screen. Exciting!



Once open, CodeBrowser will ask if you would like to analyze ch1.bin. Select Yes to view Analysis Options in the window. All analyzers are checked by default except Decompiler Parameter ID and the prototypes highlighted in light blue. Go ahead and click Analyze. Ghidra will complete auto-analysis to set up the user for manual analysis.




Take a minute to explore Ghidra windows such as Program Trees and Listing. You can customize Ghidra in whichever way suits your analysis style. The recommended setup is to have Listing and Decompiler as main windows at a minimum. These windows provide two levels of simultaneous analysis: one at assembly level and one at source code level. Notice the ELF magic number, 0x7F (followed by 0x454C46), is selected at the top of Listing and returns no function to the Decompiler. Let's check out a function to see what is returned. From CodeBrowser select Window >> Functions to retrieve a list of found functions.

Since this is a beginner challenge, we see common functions such as _init, printf, and main. Double-clicking main in the Functions window gives us interpreted C-code in the Decompiler window.
